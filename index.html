GENCODE — สรุปการเปลี่ยนแปลง

เอกสารสำหรับทีม (พี่พิม · พี่แอมแป) · เทียบของเดิม ↔ ของใหม่ v3.10DRAFTรอ Dr.Do รีวิว

หน้านี้สรุปว่า ระบบรหัส GENCODE เปลี่ยนอะไรไปบ้าง จากของเดิม — มีตารางเทียบ ชื่อไทย / อังกฤษ / รหัส และแยกชัดว่าอันไหน ของเดิม อันไหน เพิ่งเพิ่ม

1 · ภาพรวม — หลักการ 2 ชั้น (เคาะแล้ว)

ก่อนดูรหัส ขอปูภาพใหญ่ก่อน เพราะตอนนี้ขอบเขตถูกล็อกแล้ว:

ชั้นหน้าที่
GENCODEชั้น catalog — เข้ารหัสเฉพาะ สิ่งที่คลินิกทำ/ขายจริง: treatment / course / product
Transaction layerทุกอย่างเรื่องการจ่าย: วิธีจ่าย / comp / ส่วนลด / ฟรี / แต้ม / เงินเติม / influencer / ค่าสมาชิก — ไม่เข้ารหัสใน GENCODE
JERA วันนี้transaction layer ที่ใช้อยู่จริงตอนนี้
Sales-Order appเฟสถัดไปหลัง GENCODE — mimic transaction layer ของ JERA ยังไม่ใช่สิ่งที่ต้อง built ในรอบนี้

เพราะ JERA แบนราบ สิ่งที่เป็น ตัวตนของคอร์ส/สินค้า อยู่ใน GENCODE ส่วนรายละเอียดการขายอยู่ใน transaction layer แยกออกไป

ตัดสินใจแล้ว 2026-06-14: GENCODE = real treatment/course/product only · ราคา / ส่วนลด / ฟรี / แต้ม / เงินเติม / ค่าสมาชิก = transaction layer ไม่เข้ารหัส · transaction layer วันนี้ = JERA · Sales-Order app = เฟสถัดไปหลัง GENCODE

2 · อะไรเปลี่ยนไป (diff จากของเดิม)

① ตัด checksum (8→7)

รหัส CRS เหลือ 7 ช่อง · SEQ (ช่อง 6) ยังอยู่ ตัดทิ้งคือ checksum (ช่อง 8 เดิม)
8 ช่อง (มี checksum) → 7 ช่อง (SEQ คงไว้)

② SUBTYPE เพิ่ม WAR

ช่อง 3 เคยมีแค่ INI/OPT — เพิ่ม WAR (เคสประกัน ฟรี)
INI · OPT → INI · OPT · WAR

③ ศัลย์ = scheme B

ช่อง 5 = เทคนิค/ตำแหน่ง · ช่อง 7 = หมอ
ช่อง 5 = หมอ → ช่อง5=เทคนิค · ช่อง7=หมอ

④ ซิลิโคนใช้ชื่อยี่ห้อ

เลิกใช้ STD/PREM ที่อ่านไม่รู้ยี่ห้อ — ย้ายเป็น technique token ของจมูก
SLSTD/SLPREM → SILBD/SILBLK/SILIMP/SILUP

⑤ PKG = เอกสารในชั้นคอร์ส

แพกเกจ/แคมเปญ = เอกสารในชั้น Course ไม่ใช่ช่องรหัสแยก
ระบบขนาน → documentation ในชั้น CRS

⑥ เลิกขาย ≠ ตัดรหัส

ของเลิกใช้แล้วยังมี code ได้ · แยกสถานะด้วย isActive(0,1)
ตัดทิ้งตาม del_flag/show_flag → เก็บ code + isActive

3 · โครง CRS — รหัสคอร์สถาวร (7 ช่อง + ช่อง 8 ตัวเลือก)

CRS - {TYPE} - {SUBTYPE} - {CATEGORY} - {DIFFERENTIATOR} - {SEQ} - {VARIANT} [- {VERSION}]
 1      2         3            4              5             6        7         8 (ตัวเลือก)

ตัวอย่าง: CRS-SKN-INI-FIL-RST-001-4CC
          ฟิลเลอร์ Restylane 4cc (เคสรีวิว)
1 · CRS = คอร์ส 2 · TYPE = แผนก 3 · SUBTYPE = หลัก/เสริม 4 · CATEGORY = หมวด 5 · DIFFERENTIATOR = ตัวแยก 6 · SEQ = รีวิว 7 · VARIANT = รุ่นย่อย 8 · VERSION = เวอร์ชัน (ตัวเลือก)
ตัด checksum ไม่ใช่ตัด SEQ: ของเดิม 8 ช่อง ช่องที่ 8 = checksum (ตัวอักษรตรวจสอบ) — อันนั้นที่ตัดทิ้ง. SEQ (ช่อง 6 = สถานะรีวิว 001/002) ยังอยู่ในรหัส ตามเดิม. ยืนยันจาก engine จริงที่ทดสอบแล้ว: ["CRS",type,subtype,category,differentiator,seq,variant]
VERSION (ช่อง 8) ปกติไม่มี (= V001 โดยปริยาย) · dup ที่เก็บ → -V002, -V003 · แยกจาก SEQ (รีวิว) — review ซ้ำ 2 รอบก็ยังต่างกันได้

4 · WAR — เคสประกัน (subtype ใหม่) ใหม่ · lock v3.9

ของใหม่ล่าสุด: เพิ่ม WAR เป็น SUBTYPE ตัวที่ 3 (ช่อง 3) คู่กับ INI/OPT — สำหรับ เคสแก้ฟรีในช่วงรับประกันหลังผ่าตัด (warranty revision, ราคา = 0)

ทำไมต้องมี WAR แยก (ไม่ใช่แค่ตั้งราคา 0): เดิมหลักการคือ "ในประกัน = แค่ flag ราคา 0 ที่ใบขาย (Sales Order)" — แต่ Captain ตัดสินใจ (2026-06-14) ให้ encode ที่ระดับรหัส เพื่อให้ รายงานเคสประกันได้รายหมอ (back-office track ต้นทุน: เคสใหม่ vs เคสแก้เก็บเงิน vs เคสแก้ฟรี แยกตามหมอได้ที่ชั้นคอร์สเลย). เป็น การ override หลักการเดิมโดยตั้งใจ ยอมให้รหัสเพิ่มจำนวนเพื่อแลกกับรายงานที่ละเอียด
WAR vs OPT — อย่าสับสน: WAR = แก้ ฟรี ในประกัน (ราคา 0)  ·  OPT = แก้แบบ เก็บเงิน (charged revision) ที่ลูกค้าจ่ายเอง. คนละ subtype. ของแก้แบบเก็บเงินทั่วไปยังใช้ OPT เหมือนเดิม เช่น CRS-SUR-OPT-NOSE-REVIS-001-DRGP (แก้จมูกเก็บเงิน)

WAR ครอบทุกแผนก (ไม่ใช่แค่ศัลย์)

ที่ไหนมีประกัน WAR ใช้ได้หมด — ศัลยกรรม + สกิน + ฉีด:

ศัลย์ — ตาแก้ตาฟรีในประกัน (หมอไอยู)CRS-SUR-WAR-EYE-REVIS-001-DRIU

6 · รหัสศัลยกรรม (SUR scheme B) lock v3.9

ศัลยกรรมแยกตาม เทคนิค/ตำแหน่ง ไม่ใช่หมอ → ใส่ใน DIFFERENTIATOR (ช่อง 5)  ·  หมอ → VARIANT (ช่อง 7) (กรอกตอนสร้าง)

เคส "ปากซ้ำ": ศัลยกรรมปากมี 3 แบบ (ปากบน / ปากล่าง / บน+ล่าง). ถ้า differentiator = หมอ → ทั้ง 3 แบบรหัสชนกัน. แก้เป็น differentiator = เทคนิค/ตำแหน่ง (UPPER / LOWER / BOTH) → แยกกันได้สะอาด

7 · Set ผ่าตัดฆ่าเชื้อ — เป็น "คอร์ส" (ไม่ใช่ service) ใหม่ · 2026-06-14

"ชุด Set ผ่าตัดฆ่าเชื้อ" = คอร์สที่ขาย/เก็บเงินเอง (CRS) — เหมือน Set อื่นๆ ที่เป็น CRS-MSC-INI-SET-* อยู่แล้ว ไม่ใช่ service ที่ผูกอยู่ในชื่อคอร์สอื่น

แก้ความเข้าใจเดิม: ก่อนหน้านี้ GLUEBOY แมป Set ฆ่าเชื้อไว้ผิด — เอาไปอยู่ชั้น service + ฝังในชื่อคอร์ส. ฝน (data partner) ชี้ว่าจริงๆ มันคือ ของที่ลูกค้าจ่ายเพิ่ม → ต้องเป็น CRS (คอร์ส) ของตัวเอง

8 · Dedup + เปิด/ปิด (active)

เมื่อระบบเจอรหัสที่ "ออกมาเหมือนกัน" ต้องมีนโยบายจัดการ — และต้องแยก ราคา ออกจาก ตัวรหัส ให้ชัด:

dedup policy:
เจอรหัสซ้ำ (คนละราคา) → flag ถาม user
   ├ เก็บ (ของจริงคนละตัว)  → +V002  (กลายเป็น 2 รหัส)
   └ ผิด (คีย์พลาด)         → soft-delete ตัวเดิม + สร้างใหม่
price = transaction (ไม่อยู่ในรหัส). "รหัสเดียว ราคาต่าง" เป็น SIGNAL ให้เช็ก 3 อย่าง:
① แคมเปญ → ไปอยู่ที่ transaction  ·  ② attribute ขาด → เพิ่ม token  ·  ③ dup → ลบ.
ราคา tier ที่ มีความหมาย = ทำเป็น token (เช่น silicone SILBD / SILBLK / SILIMP)
active / soft-delete = isActive(0,1) (Nothing-is-Deleted): รหัส = ตัวตนถาวร, active = สถานะ ไม่ใช่ตัวกรอง — ปิดการขายได้ แต่รหัสไม่หาย

9 · PKG — แคมเปญ / แพกเกจ / ซับแพกเกจ

ชั้นบางๆที่ อ้างถึง รหัส CRS (ไม่ใช่ระบบขนาน):

⚠️ ตำแหน่งของ PKG (Captain เคาะ 2026-06-14): PKG (แคมเปญ/แพกเกจ) อยู่ในชั้น Course เป็น "เอกสาร" (documentation) — เนื้อหา PKG-CODE-DRAFT.md ยุบรวมเข้า CRS-CODE.md ไฟล์เดียว (ลด spec file, ตาม SSOT). PKG ไม่ใช่ "ช่อง" ในรหัส 7 ช่อง และ ไม่ใช่ layer แยกขนาน — โครงรหัส 7 ช่องไม่เปลี่ยน. แคมเปญที่เป็น bundle จริงยังเข้ารหัสได้ที่ VARIANT (เช่น C2602P1) ตามสเปกเดิม
⚠️ อย่าสับสน 2 แกน: INI/OPT = property ของ คอร์ส (หลักเริ่มต้น vs ออพชั่น/ตัวเสริม)  ·  package/subpackage = โครงของ แคมเปญ (ชุดฐาน vs ตัวอัพในชุด) — คนละแกน ไม่เท่ากัน. แคมเปญหยิบ "คอร์ส" (ที่เป็น INI หรือ OPT) มาประกอบเป็น package/subpackage
PKG-{GROUP}-{HOOK}-{C+YYMM}-{SET}          ← แพกเกจหลัก
PKG-{GROUP}-{HOOK}-{C+YYMM}-{SET}-{UPGRADE} ← ซับแพกเกจ (อัพ)

ตัวอย่าง: PKG-SKN-OLG-C2602-SETA-U400

  C2602 = แคมเปญ ปี2026 เดือน02 (C=campaign)
  SETA  = ชื่อชุด (คนตั้งเอง)
  U400  = อัพเป็นอะไร (ตั้งชื่อตามสิ่งที่อัพ)

เข้าใจ 3 ระดับให้ชัด (นี่คือของใหม่ที่สุด)

แคมเปญ (campaign)  ─ ห่อด้วย เดือน-ปี + เงื่อนไข
 └ แพกเกจ (package) ─ ชุดฐาน = สินค้าหลัก(hook) + ของแถม
     ├ ซับแพกเกจ #1 ─ ตัวเลือกอัพ (เช่น อัพชอต)
     └ ซับแพกเกจ #2 ─ ตัวเลือกอัพ (เช่น เปลี่ยนยี่ห้อ)

12 · Map Logic — เครื่องแปลงชื่อไทย → รหัส สร้างแล้ว ✓

นี่คือ "สมอง" ที่ทำให้ทีม คีย์คอร์สใหม่ได้เอง โดยไม่ต้องให้ Dr.Do แมปทีละตัว — พิมพ์ชื่อไทย → ได้รหัส GENCODE อัตโนมัติ และถ้าไม่แน่ใจมัน หยุดถาม ไม่เดามั่ว

สร้าง + พิสูจน์ในรีโป YD_GENCODE ก่อน (เป็น pure function รันได้เลย ไม่ต้องต่อ DB) แล้วค่อยพอร์ตเข้า engine จริง (glyph) — ปลอดภัย ไม่เสี่ยง production · เห็นผลทันที

Logic — ลำดับการเรียกฟังก์ชันจริง (ชื่อไทย → รหัส)

ขั้น 1 · wizard เก็บ context (user เลือก)
   กลุ่ม→TYPE · แผนก→CATEGORY · หลัก/เสริม→SUBTYPE · (ศัลย์)หมอ→DOCTOR · พิมพ์ชื่อ→name

ขั้น 2 · เรียก parser (ฟังก์ชันเดียวที่ UI เรียก)
   nameToSkinSegments(name, type, category, subtype)          ← parser.ts
   (ศัลย์ = nameToSegments(name, category, subtype, doctor))

ขั้น 3 · ข้างใน parser เรียกต่อตามนี้
   6a  skinTokens(type, category)        → registry-skin.ts   ดึง list token ของหมวด
   6b  matchToken(name, tokens)          → match.ts           จับคู่ keyword → token (ช่อง 5)
          └ ไม่เจอ / เสมอ → ⚑ FLAGGED (จบ → ขั้น 4 ถามคน)
   6c  seqOf(name)                       → parser.ts          "ไม่รีวิว"? → 002 ไม่งั้น 001 (ช่อง 6)
   6d  extractVariant(name, cat, token)  → variant.ts         ดึงปริมาณ → 100U (ช่อง 7)
          (ศัลย์ข้ามขั้นนี้: ช่อง 7 = doctor ที่เลือกมาแล้ว)
   6e  assembleCrs(segments)             → engine.ts          ประกอบ 7 ช่อง → string
   6f  return ParseResult {code, token, confidence, flagged}

ขั้น 4 · หลังได้ผล
   ไม่ flag → โชว์ code ใน UI
   dedup: code ซ้ำของเดิม? → "ซ้ำคนละราคา เก็บไหม?" → เก็บ = -V002 / ไม่ = soft-delete ตัวเก่า

ตัวอย่างจริง — trace 2 เคส (เห็นจุดต่าง ศัลย์/สกิน)

สกิน · "โบท็อกซ์ แอลเลอร์แกน 100 ยูนิต" (SKN / BTX / INI)

6b matchToken → "แอลเลอร์แกน" = ALG (ไม่มีคำโซน buffet) → token = ALG

ศัลย์ · "เสริมจมูกใหม่ เซมิโอเพ่น(Semi-open)" (SUR / NOSE / INI / หมอ Do)

6a SUR_REGISTRY[NOSE][INI] → [OPEN, SEMI, CLOSED]
6b matchToken
     "เซมิ"   = keyword ของ SEMI (priority 10)
     "โอเพ่น" = keyword ของ OPEN (priority 0)   ← ชื่อมีทั้งคู่!
     → SEMI ชนะเพราะ priority สูงกว่า → token = SEMI
6c seqOf   → 001
6d ⚠️ ศัลย์ข้าม extractVariant → ช่อง 7 = หมอ = DRDO

3 จุดที่ trace นี้สอน: ① priority แก้คำกำกวม — "เซมิโอเพ่น" มีคำว่า "โอเพ่น" อยู่ด้วย แต่ SEMI priority สูงกว่า เลยไม่หลงไป OPEN  ·  ② ศัลย์: ช่อง 7 = หมอ (จาก wizard) ไม่ดึงจากชื่อ ต่างจากสกินที่ดึงปริมาณ  ·  ③ case-insensitive — "Semi-open" (S ใหญ่) ก็จับ keyword "semi" ได้

13 · UX/UI Spec สำหรับ Pym DRAFT v0.1

DRAFT v0.1 — for owner + Pym review. หัวข้อนี้เป็น requirements spec ว่า UI ที่ขับด้วย GENCODE ต้องทำอะไรบ้าง ไม่ใช่ mockup และไม่ใช่ visual direction. Pym ออกแบบ flow, screen, interaction detail, และ visual hierarchy ได้เต็มที่ภายใต้โครง GENCODE ที่ fixed แล้ว

Source of truth: โครงรหัสและตัวอย่างในสเปกนี้อ้างจาก YD_GENCODE/CRS-CODE.md v3.9, README layer summary, และ mapper/ ที่ทดสอบชื่อไทย → GENCODE ได้ 340/340 gold. หน้านี้ชี้ไปที่ catalog ไม่คัดลอก token list เต็ม เพื่อกัน drift

ขอบเขตที่ fixed vs สิ่งที่ Pym ออกแบบได้

Fixed by GENCODEPym designs

สามชั้นหลักยังแยกหน้าที่: CRS = สิ่งที่ลูกค้าซื้อ, SVC = สิ่งที่ทำจริง, PRD = ของ/สต็อกที่ใช้จะโชว์ layer ไหนใน order-entry, breakdown, drawer, หรือ review step แบบไหนให้ทีมเข้าใจง่าย
Scheme B สำหรับ SUR: ช่อง 5 = technique/part, ช่อง 7 = doctor. ตัวอย่าง verified: CRS-SUR-INI-NOSE-SEMI-001-DRDOวิธีให้เลือกหมอ, วิธีอธิบายความต่าง surgery vs skin, และวิธีลดความสับสนของ operator
mapper ต้อง flag เมื่อไม่รู้หรือกำกวม. Low confidence ห้าม auto-write ลง catalogรูปแบบ review/confirm, microcopy, severity, และวิธีส่งงานให้ owner ratify
ราคา, วิธีจ่าย, comp, ส่วนลด, ฟรี, แต้ม, เงินเติม, influencer, ค่าสมาชิก, cashier, branch, และ transaction fact ไม่ใช่โครง CRS catalog codeจะวาง transaction fields ใกล้ code breakdown แค่ไหน เพื่อให้ทีมเห็นว่าเป็น sales-order data ไม่ใช่ segment ใหม่

Surface ① — Thai name → GENCODE search

นี่คือ core surface สำหรับ operator: พิมพ์หรือค้นชื่อบริการภาษาไทย แล้วระบบคืนรหัส GENCODE พร้อมสถานะความมั่นใจ ก่อนให้ผู้ใช้ยืนยัน

Input:
  Thai service/course name
  required context: TYPE, CATEGORY, SUBTYPE
  surgery-only context: DOCTOR

Mapper output:
  code / segments / matched token / confidence / flagged / reason / candidates

Decision:
  high confidence + not flagged  → operator confirms → usable code
  low or none + flagged          → stop → human picks/ratifies → save alias only after confirmation
StateRequirementWhy
Exact/high confidenceShow the generated code, matched segment labels, and a clear confirm action. Example verified by mapper tests: CRS-SKN-INI-BTX-ALG-001-100U for Allergan botox 100 unitsOperator should trust but still see what the parser did before order/catalog write
Ambiguous/low confidenceShow that the mapper found competing candidates. Require a human choice; do not silently prefer one token just because it can assemble a codeThe matcher intentionally reports true ties as ambiguous, because wrong codes are harder to clean later
No matchShow the reason, preserve the typed name, and offer the code-builder path or an owner-ratification queueUnknown brand/procedure must become a reviewed alias or catalog gap, not an invented code
Confirmed aliasOnce owner/authorized user confirms, store name → token as an alias so the same spelling resolves automatically next timeThe assist flow is confirm-once, remember-after; deterministic parser stays the trust anchor
Low-confidence rule: UI must stop before write. It may suggest candidates, but the action is review / choose / ratify, not auto-save. The visible status should make the next owner action obvious: confirm existing token, open code-builder, or mark a YD_GENCODE gap

Surface ② — Cascading code-builder

ใช้เมื่อ name-search flag, เมื่อสร้าง catalog item ใหม่, หรือเมื่อ operator รู้ segment ที่ต้องการมากกว่าชื่อไทย. Builder ไม่ใช่ token catalog copy; มันควรดึง allowed values จาก registry/source จริง

TYPE
  → SUBTYPE
    → CATEGORY
      → DIFFERENTIATOR
        → SEQ
          → VARIANT
            → optional VERSION only if duplicate policy says keep
StepRequirement
TYPEเลือก department/business unit ก่อน เช่น SUR/SKN/VIT/TRT/MSC; downstream choices filter immediately
SUBTYPEเลือก role ของคอร์ส เช่น INI/OPT/WAR ตาม source. ถ้า engine integration ยังไม่รองรับค่าใด ต้องแสดงเป็น implementation gap ไม่ใช่ซ่อนใน UI
CATEGORYเลือกหมวดจาก TYPE ที่เลือก. UI ต้อง point to YD_GENCODE catalog แทนการฝัง list ถาวรในหน้า design
DIFFERENTIATORความหมายขึ้นกับ category profile. สำหรับ SUR = technique/part; สำหรับ skin/device หลายหมวด = brand/device/area/method
SEQเลือก review status: 001 review, 002 not-review. UI ต้องไม่สับสนกับ optional VERSION
VARIANTสำหรับ SUR = doctor; สำหรับ non-SUR = amount เช่น cc/units/shots/sessions ตาม profile
Builder feedback: ระหว่างเลือก ให้แสดง code preview แบบโตทีละ segment และแสดง human-readable breakdown ข้างกัน. ถ้าเลือก combination ที่ registry ไม่มี ให้ flag เป็น "needs ratification" ไม่ให้สร้าง token ใหม่เอง

Surface ③ — Order-entry flow + validation

Order-entry ต้องช่วยทีมเลือกสิ่งที่จะขายโดยไม่ปน catalog identity กับ transaction facts. การสร้างหรือเลือก code ควรจบด้วย validation + breakdown ก่อนบันทึก order

1. Start order
2. Search Thai name or open code-builder
3. Select/confirm CRS code
4. Validate: exists, active, source-backed, non-flagged, duplicate policy resolved
5. Show code breakdown in human terms
6. Add sales-order facts: price, campaign, branch, cashier, customer, payment, notes
7. Save order with CRS reference; SVC/PRD detail remains separate by layer
ValidationRequired behavior
Source-backed codeEvery displayed code must resolve to YD_GENCODE mapper/spec. If not, show "gap for ratification" and block normal save
Duplicate/collisionIf generated code already exists with different price/details, ask keep vs error. Keep appends VERSION; error path soft-deletes/replaces per policy
Active statusInactive/soft-deleted codes can be shown for audit, but require explicit override or cannot be selected for a new order depending on owner policy
Layer boundaryPrice/campaign/reviewable sales context must be shown as order fields unless the fixed CRS spec says it is a segment. Do not let screen design imply a new segment
Breakdown viewAlways read the code back: prefix, type, subtype, category, differentiator, seq, variant, optional version, plus source/flag status

Code-breakdown requirements

Breakdown is the trust surface. It lets an operator see that the code means what they are about to sell, and it gives reviewer/owner a compact audit target

Example breakdown source: CRS-CODE.md v3.9
CRS-SKN-INI-FIL-RST-001-4CC
  CRS  = course
  SKN  = skin
  INI  = initial/main course
  FIL  = filler
  RST  = Restylane differentiator
  001  = review
  4CC  = 4cc variant
Design intent: Pym can decide whether this appears as a panel, drawer, review step, or inline inspector. The requirement is that the operator can inspect the generated code before save, and flagged/low-confidence cases cannot look visually equivalent to verified cases

Open review points for owner + Pym

  • Who is allowed to ratify a flagged token or alias: owner only, manager, or selected operator role?
  • Should order-entry default to name-search first, or should high-risk categories like SUR start with builder context first?
  • Where should the review queue live: inline in order-entry, a separate catalog admin surface, or both?
  • How much source detail should operator UI show: source path only, source version, or full audit trail?
