GENCODE — สรุปการเปลี่ยนแปลง

เอกสารสำหรับทีม (พี่พิม · พี่แอมแป) · เทียบของเดิม ↔ ของใหม่ DRAFTรอ Dr.Do รีวิว

หน้านี้สรุปว่า ระบบรหัส GENCODE เปลี่ยนอะไรไปบ้าง จากของเดิม — มีตารางเทียบ ชื่อไทย / อังกฤษ / รหัส และแยกชัดว่าอันไหน ของเดิม อันไหน เพิ่งเพิ่ม

1 · ภาพรวม — ระบบทำงาน 3 ชั้น

ก่อนดูรหัส ขอปูภาพใหญ่ก่อน เพราะมันเปลี่ยนวิธีคิดนิดนึง:

ชั้นหน้าที่

JERAชั้นใช้งานจริง — แบนราบ 1 แถวต่อ 1 สิ่งที่ขาย ไม่มีลำดับชั้น

เพราะ JERA แบนราบ ลำดับชั้นที่มันเก็บไม่ได้ เลยไปอยู่ ในตัวรหัส และการจัดกลุ่มแคมเปญอยู่ใน registry ของ GENCODE

หลักการ 2 ชั้น: รหัส = ตัวตนของสินค้า/คอร์ส (มันคืออะไร) · ราคา / ส่วนลด / แคมเปญที่ลูกค้าเลือก = transaction (การขาย) — ไม่ปนกัน

3 · โครง CRS — รหัสคอร์สถาวร (7 ช่อง + ช่อง 8 ตัวเลือก)

CRS - {TYPE} - {SUBTYPE} - {CATEGORY} - {DIFFERENTIATOR} - {SEQ} - {VARIANT} [- {VERSION}]
 1      2         3            4              5             6        7         8 (ตัวเลือก)

ตัวอย่าง: CRS-SKN-INI-FIL-RST-001-4CC
          ฟิลเลอร์ Restylane 4cc (เคสรีวิว)
1 · CRS = คอร์ส 2 · TYPE = แผนก 3 · SUBTYPE = หลัก/เสริม 4 · CATEGORY = หมวด 5 · DIFFERENTIATOR = ตัวแยก 6 · SEQ = รีวิว 7 · VARIANT = รุ่นย่อย 8 · VERSION = เวอร์ชัน (ตัวเลือก)

5 · รหัสศัลยกรรม (SUR scheme B)

ศัลยกรรมแยกตาม เทคนิค/ตำแหน่ง ไม่ใช่หมอ → ใส่ใน DIFFERENTIATOR (ช่อง 5)  ·  หมอ → VARIANT (ช่อง 7) (กรอกตอนสร้าง)

เคส "ปากซ้ำ": ศัลยกรรมปากมี 3 แบบ (ปากบน / ปากล่าง / บน+ล่าง). ถ้า differentiator = หมอ → ทั้ง 3 แบบรหัสชนกัน. แก้เป็น differentiator = เทคนิค/ตำแหน่ง (UPPER / LOWER / BOTH) → แยกกันได้สะอาด
ตัวอย่าง (differentiator = เทคนิค/ตำแหน่ง · หมอ = VARIANT):

6 · Dedup + เปิด/ปิด (active)

เมื่อระบบเจอรหัสที่ "ออกมาเหมือนกัน" ต้องมีนโยบายจัดการ — และต้องแยก ราคา ออกจาก ตัวรหัส ให้ชัด:

dedup policy:
เจอรหัสซ้ำ (คนละราคา) → flag ถาม user
   ├ เก็บ (ของจริงคนละตัว)  → +V002  (กลายเป็น 2 รหัส)
   └ ผิด (คีย์พลาด)         → soft-delete ตัวเดิม + สร้างใหม่
price = transaction (ไม่อยู่ในรหัส). "รหัสเดียว ราคาต่าง" เป็น SIGNAL ให้เช็ก 3 อย่าง:
① แคมเปญ → ไปอยู่ที่ transaction  ·  ② attribute ขาด → เพิ่ม token  ·  ③ dup → ลบ.
ราคา tier ที่ มีความหมาย = ทำเป็น token (เช่น silicone BDEV / MKA / IMPL)
active / soft-delete = หน้าที่ user (Nothing-is-Deleted): รหัส = ตัวตนถาวร, active = สถานะ ไม่ใช่ตัวกรอง — ปิดการขายได้ แต่รหัสไม่หาย

7 · ชั้น PKG ใหม่ — แคมเปญ / แพกเกจ / ซับแพกเกจ

ของใหม่ล้วน — ชั้นบางๆที่ อ้างถึง รหัส CRS (ไม่ใช่ระบบขนาน):

⚠️ อย่าสับสน 2 แกน: INI/OPT = property ของ คอร์ส (หลักเริ่มต้น vs ออพชั่น/ตัวเสริม)  ·  package/subpackage = โครงของ แคมเปญ (ชุดฐาน vs ตัวอัพในชุด) — คนละแกน ไม่เท่ากัน. แคมเปญหยิบ "คอร์ส" (ที่เป็น INI หรือ OPT) มาประกอบเป็น package/subpackage
PKG-{GROUP}-{HOOK}-{C+YYMM}-{SET}          ← แพกเกจหลัก
PKG-{GROUP}-{HOOK}-{C+YYMM}-{SET}-{UPGRADE} ← ซับแพกเกจ (อัพ)

ตัวอย่าง: PKG-SKN-OLG-C2602-SETA-U400
  SKN   = กลุ่ม (skin/surgery)
  OLG   = HOOK สินค้าหลัก/ตัวดึงดูด
  C2602 = แคมเปญ ปี2026 เดือน02 (C=campaign)
  SETA  = ชื่อชุด (คนตั้งเอง)
  U400  = อัพเป็นอะไร (ตั้งชื่อตามสิ่งที่อัพ)
ระดับชื่อไทยEnglishรหัส
packageชุดแคมเปญ (ชุดฐาน)packagePKG-…-SET
subpackageตัวอัพในชุดsubpackagePKG-…-SET-U…
campaignแคมเปญลงวันที่campaignPKG-…-C2602
อะไรอยู่ตรงไหน: ในรหัส = กลุ่ม·hook·เดือน·ชุด·อัพ  |  ในนิยาม (registry) = BOM ของแถม·ชื่อไทยเต็ม·เงื่อนไข  |  ใน transaction = ราคา·ส่วนต่าง·อัพที่ลูกค้าเลือก

เข้าใจ 3 ระดับให้ชัด (นี่คือของใหม่ที่สุด)

package เป็นของใหม่ที่สุดในระบบ — ขออธิบายละเอียด. มันคือ "ชุดที่ขาย" ซึ่งประกอบจาก คอร์สหลายตัว มาขายเป็นแพ็ค พร้อมตัวเลือกอัพเกรด:

แคมเปญ (campaign)  ─ ห่อด้วย เดือน-ปี + เงื่อนไข
 └ แพกเกจ (package) ─ ชุดฐาน = สินค้าหลัก(hook) + ของแถม
     ├ ซับแพกเกจ #1 ─ ตัวเลือกอัพ (เช่น อัพชอต)
     └ ซับแพกเกจ #2 ─ ตัวเลือกอัพ (เช่น เปลี่ยนยี่ห้อ)
         ▲ ทุกชั้น "อ้างถึง" คอร์ส CRS จริง (ไม่ทำซ้ำ)
ระดับคืออะไร (ละเอียด)ตัวอย่าง
package
แพกเกจ		ชุดฐานที่ขาย = สินค้าหลัก (hook = ตัวดึงดูด/มูลค่าสูงสุด) + ของแถม/เสริม (BOM) ขายเป็นชุดราคาโปร"โปรย้อนวัย SETA" = Oligio 300sh + แถม Ultraformer + fat + Gluta
subpackage
ซับแพกเกจ		ตัวเลือก อัพเกรด ที่แตกจาก package หลัก — ลูกค้าจ่ายเพิ่มเพื่ออัพอัพ Oligio 300→400 shots (จ่ายส่วนต่าง)
campaign
แคมเปญ		ตัวห่อ package ลงช่วงเวลา (เดือน-ปี) + เงื่อนไข (อัพในช่องสีเดียวกัน · ลดลูกค้าเก่า)แคมเปญเดือน ก.พ. 2026 = C2602

ซับแพกเกจ (อัพเซลล์) มี 4 แบบ

แบบคือตัวอย่างท้ายรหัส
tier-jumpขยับขั้น/จำนวน-U400 (300→400sh)
brand-swapเปลี่ยนยี่ห้อ-BNX50 (เปลี่ยนเป็น Bienox)
qty-addเพิ่มจำนวน-ADD5 (เพิ่ม fat 5cc)
exchange-buyแลกซื้อ-X12 (แลกซื้อ 12 ครั้ง)
หัวใจ: รหัส package/subpackage เก็บแค่ โครงสร้าง (อะไรอยู่ในชุด · อัพเป็นอะไร) — ราคา/ส่วนต่าง/แคมเปญที่ลูกค้าเลือก = transaction (ไม่ฝังในรหัส) → เปลี่ยนราคา/ทำโปรใหม่ได้ โดยรหัสสินค้าไม่เปลี่ยน

10 · Map Logic — เครื่องแปลงชื่อไทย → รหัส สร้างแล้ว ✓

นี่คือ "สมอง" ที่ทำให้ทีม คีย์คอร์สใหม่ได้เอง โดยไม่ต้องให้ Dr.Do แมปทีละตัว — พิมพ์ชื่อไทย → ได้รหัส GENCODE อัตโนมัติ และถ้าไม่แน่ใจมัน หยุดถาม ไม่เดามั่ว

สร้าง + พิสูจน์ในรีโป YD_GENCODE ก่อน (เป็น pure function รันได้เลย ไม่ต้องต่อ DB) แล้วค่อยพอร์ตเข้า engine จริง (glyph) — ปลอดภัย ไม่เสี่ยง production · เห็นผลทันที

Logic — ชื่อไทยกลายเป็นรหัสยังไง (flow)

ชื่อไทย ──► parser ──► CRS-{TYPE}-{SUB}-{CAT}-{DIFFERENTIATOR}-{SEQ}-{VARIANT}

wizard เก็บ context (user เลือก):
  แผนก(TYPE) · หมวด(CAT) · หลัก-เสริม(SUBTYPE) · หมอ(เฉพาะศัลย์)
ชื่อหัตถการ (พิมพ์) ──┐
                     ▼
1  registry    ดึง list token + keyword ของหมวดนั้น
2  matchToken  จับคู่ keyword → priority สูงก่อน → คำยาวสุด → ได้ token (ช่อง 5)
                 └ ไม่เจอ / ก้ำกึ่ง → ⚑ FLAG ถามคน (ไม่เดา)
3  seqOf       มีคำ "ไม่รีวิว"? → 002 · ไม่งั้น → 001 (ช่อง 6)
4  variant     ช่อง 7:  ศัลย์ = หมอ · สกิน = ดึงปริมาณจากชื่อ (100U / 2CC / 300S …)
5  assemble    ประกอบ 7 ช่อง → รหัส ✓

ตัวอย่างจริง (2 เคส)

"โบท็อกซ์ แอลเลอร์แกน 100 ยูนิต"   (สกิน / BTX / หลัก)
   → แบรนด์ ALG  +  ปริมาณ 100U
   → CRS-SKN-INI-BTX-ALG-001-100U ✓

"เสริมจมูกใหม่ เซมิโอเพ่น"          (ศัลย์ / NOSE / หลัก / หมอ Do)
   → เทคนิค SEMI (priority ชนะ OPEN ที่ก็ตรงคำว่า "โอเพ่น" เหมือนกัน)
   → หมอ = DRDO (ช่อง 7)
   → CRS-SUR-INI-NOSE-SEMI-001-DRDO ✓

ความต่าง ศัลย์ vs สกิน

ช่อง 5 (ตัวแยก)ช่อง 7 (variant)
ศัลย์เทคนิค / ตำแหน่งหมอ (เลือกตอนสร้าง)
สกินแบรนด์ / เครื่อง / โซนปริมาณ (ดึงจากชื่อ)
